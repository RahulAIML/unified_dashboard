\section{Executive Summary}
% =============================================================================

\begin{infobox}
\textbf{Key finding:} The database is operational and queryable.  All data
originates from a single 9-day window (Apr 7--15, 2026) across 19 unique use
cases.  Two tables store all analytics in an Entity-Attribute-Value (EAV)
structure.  The system is \textbf{live but young} --- session volumes are low
and several structural inconsistencies (mixed score scales, absent pass/fail
fields) must be resolved before global KPI aggregation is valid.
\end{infobox}

\vspace{0.5cm}

\subsection{Entity-Attribute-Value Design}

\begin{infobox}
\textbf{Design pattern:} Every session (\texttt{saved\_report\_id}) generates
N rows in \texttt{report\_field\_current} --- one row per metric.  Different
use cases produce different sets of \texttt{field\_key} values.  There is
\textbf{no fixed schema per session}; the fields present depend entirely on
the use case that generated the report.
\end{infobox}

\textbf{Row counts per session by use case (sample):}
Use case 401 produces the most rows per session ($\approx$35 rows/session),
while use cases 369, 400, and 402 produce only 7--12 rows/session.

\section{Critical Gaps and Blockers}
\label{sec:gaps}
% =============================================================================

\begin{warnbox}
\textbf{These gaps must be resolved before building a production dashboard
with aggregated global KPIs.}
\end{warnbox}

\subsection{Gap 1 --- Mixed Score Scales (Blocker for Global Avg Score)}

\textbf{Resolution required:} Either (a) normalise all scores to 0--100 in the
API layer, or (b) always show scores per-usecase and never aggregate across
different use cases.

\subsection{Gap 2 --- Pass/Fail Only Present for 3 Use Cases}

\texttt{overall\_result} exists in only 14 of 77 total sessions (use cases 363, 381,
394).  The remaining 63 sessions have \textbf{no pass/fail outcome}.
Consequently:

\begin{itemize}[leftmargin=*]
  \item A global pass rate metric cannot be computed honestly.
  \item Pass rate is valid only for \textbf{Certification (UC 394)}: 7 Básico + 3 Bueno + 1 Deficiente out of 11 sessions = \textbf{90.9\%} pass.
  \item Coach UC 381: 1 Bueno, 1 Básico (2 sessions only --- statistically insignificant).
\end{itemize}

\subsection{Gap 3 --- No User Identifier}

Neither table contains a user ID, user name, or email column.  It is
\textbf{impossible} to build per-user KPIs, leaderboards, or progress tracking
from the current schema.  This is a fundamental limitation for any
``individual performance'' dashboard feature.

\subsection{Gap 4 --- Use Case 401 Not Mapped to a Module (Now Fixed)}

UC 401 is the \textbf{single richest data source} (15 sessions, 44 fields,
525 rows, full coaching methodology scoring).  It has now been added to the
\texttt{coach} entry in \texttt{SOLUTION\_USECASE\_MAP}.  UC 387 (55 unique
fields, 1 session) and UC 354 (71 unique fields, 1 session) still fall outside
the map.

\subsection{Gap 5 --- Very Short Data History (9 Days)}

All data spans April 7--15, 2026.  There are only \textbf{6 distinct dates}
with sessions.  Any trend chart will show at most 6 data points regardless of
the date range selected (7d, 30d, 90d all return identical data).  Meaningful
trend analysis requires at least 30 days of continuous session data.

\subsection{Gap 6 --- Zero-score Records}

Several sessions record \texttt{overall\_score = 0} (use cases 323, 369).
These may represent incomplete or failed evaluations.  Without a status field
it is impossible to distinguish ``genuinely scored 0'' from ``evaluation did
not complete.''

  \item \textbf{Add UC 401 to the solution map.} \good{Done.}

    entry in \texttt{SOLUTION\_USECASE\_MAP} and all-session query logic has
    been fixed so sessions without \texttt{overall\_score} are still counted.

  \item \textbf{Normalise score display.}
    In the API layer, detect whether \texttt{MAX(value\_num)} for a use case
    is $> 10$ (0--100 scale) or $\leq 10$ (0--10 scale).  Do not aggregate
    across scales.

  \item \textbf{Filter out zero scores.}
    Add \texttt{AND value\_num > 0} to average calculations to exclude
    incomplete evaluations that record a zero score.

  \item \textbf{Remove global pass rate KPI from overview.}
    Replace it with per-module pass rates.  Show pass rate only for modules
    where \texttt{overall\_result} is present (Certification, Coach UC 381/363).

  \item \textbf{Identify unmapped use cases (387, 354, 205).}
    Retrieve a sample report payload for each to determine their module type.

  \item \textbf{Request user\_id field from Rolplay AI.}
    Per-user analytics require a \texttt{user\_id} to be written into
    \texttt{report\_field\_current} at session creation time.  This is a
    data-model change on the AI platform side.

  \item \textbf{Build session volume before comparing periods.}
    The 7d/30d/90d filter is currently meaningless --- all data falls within
    the same 9-day window.  Trend charts will only become useful after
    30+ days of production data has accumulated.

\end{enumerate}
